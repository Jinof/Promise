# Promise Standard

这份文档是 Promise 范式的规范文本。

它的目标不是解释灵感来源，而是固定一套面向 AI Coding 的、可复用、可检查、可治理的标准。

## 1. 目的

Promise 用来替代传统 PRD，成为实现治理的上位工件。

Promise 的核心目标是：

- 先定义系统真相
- 再定义系统行为边界
- 再定义兑现证明
- 最后让代码和其它产物从 Promise 派生

Promise 不把“代码写完”视为完成。

Promise 只把“承诺被实现并有证据证明”视为完成。

## 2. 规范用语

本文中的规范词使用以下含义：

- `MUST`：必须满足，否则视为不符合 Promise 标准
- `MUST NOT`：绝对禁止
- `SHOULD`：强烈建议；如不满足，需要明确理由
- `MAY`：可选

## 3. 核心公理

Promise 范式建立在以下公理上：

1. 一个系统 `MUST` 只有一个逻辑上的 `System Promise`。
2. 系统真相优先于系统行为。
3. 系统行为优先于派生产物。
4. 验证层不创造新的系统真相，只定义兑现证明义务。
5. 派生产物 `MUST NOT` 扩张 Promise 之外的业务语义。
6. 未被证据证明的承诺，不应被视为已兑现。

## 4. 标准关系

Promise 范式包含三种固定关系。

### 4.1 逻辑唯一性

`one system -> one promise graph`

这意味着：

- 一个系统只能有一个逻辑上的语义源
- 这个语义源可以物理拆成多个模块、区块或文件
- 这些物理片段 `MUST` 最终汇总成唯一的 Promise graph
- 下游工具 `MUST` 只以这个 Promise graph 为准

### 4.2 治理层级

`字段层 > 功能层 > 派生产物`

这意味着：

- 功能层 `MUST NOT` 突破字段层的边界
- 派生产物 `MUST NOT` 突破功能层的边界
- 任何新增业务状态 `MUST` 先进入字段层

### 4.3 交付链路

`产品意图 -> System Promise -> 派生产物 -> 验证证据`

在 `System Promise` 内部，推荐编写顺序是：

`字段层 -> 功能层 -> 验证层`

这意味着：

- 先定义真相
- 再定义行为
- 再定义证明义务
- 然后生成代码、测试、Schema、接口契约等派生产物
- 最后执行验证并收集证据

## 5. 标准构成

一个完整的 `System Promise` `MUST` 是一个逻辑上的 Promise graph，并至少包含以下三个层面：

- 字段层
- 功能层
- 验证层

代码实现不是 Promise 层，而是 Promise 的派生产物。

测试、Schema、迁移、接口契约、Agent 任务也不是 Promise 层，而是 Promise 的派生产物。

## 6. 字段层标准

字段层是 `System Promise` 的最高优先级部分。

字段层 `MUST` 定义：

- 核心对象
- 字段
- 状态
- 字段语义
- 不变量
- 全局约束
- 禁止隐含状态

这些要求是字段层的覆盖义务，不应被理解为“每个 field block 都必须机械补齐固定数量的 clause”。

工具 `SHOULD` 优先检查对象是否缺少有效约束覆盖，而不是逼迫作者为了过检查补充空泛 invariant 或 forbid。

字段层 `MUST NOT` 混入以下内容：

- 页面流程
- 接口细节
- 工程实现方案
- 框架偏好

### 6.1 字段标准

每个字段 `MUST` 至少说明：

- 字段名
- 类型
- 是否必填
- 是否可空
- 默认值
- 业务语义

只有类型，没有语义，不足以满足字段层标准。

### 6.2 状态标准

只要对象存在业务状态，字段层 `MUST` 显式列出：

- 允许状态
- 每个状态的含义
- 状态迁移规则
- 终态与非终态

未列出的状态 `MUST NOT` 在派生产物中出现。

### 6.3 不变量标准

字段层 `MUST` 列出关键不变量。

不变量是指在任何合法实现和任何合法时刻都必须成立的真相，例如：

- `status = done` 时 `completedAt` 必须存在
- `status = todo` 时 `completedAt` 必须为 `null`

无法表达不变量的字段定义，不足以支撑 AI Coding。

### 6.4 禁止隐含状态

字段层 `MUST` 明确禁止以下情况：

- 用额外布尔值表达已存在状态
- 用缓存标记表达真实业务阶段
- 用临时分支表达未声明的业务语义
- 用注释、魔法值或命名约定承载业务状态

## 7. 功能层标准

功能层用来定义系统行为。

功能层 `MUST` 基于字段层编写。

功能层 `MUST` 定义：

- 行为名称
- 触发条件
- 前置条件
- 读取边界
- 写入边界
- 成功结果
- 失败或拒绝条件
- 禁止行为

这些要求是功能层的覆盖义务，不应被实现成“每个 function block 必须为了通过检查而补一条模板化 forbid”。

工具 `SHOULD` 优先检查行为边界是否存在真实漂移风险，再决定是否发出缺失告警。

### 7.1 行为边界

每个行为 `MUST` 明确：

- 允许读取哪些字段
- 允许写入哪些字段
- 不允许修改哪些字段
- 是否允许副作用

如果某个行为需要新字段或新状态，该行为 `MUST NOT` 直接扩展自身边界，而 `MUST` 先回到字段层。

### 7.2 冲突处理

当功能层与字段层冲突时：

- 字段层 `MUST` 优先
- 功能层 `MUST` 被修订
- 派生产物 `MUST` 服从修订后的 Promise

### 7.3 禁止事项

功能层 `MUST NOT`：

- 引用未声明字段
- 引用未声明状态
- 隐式扩大读写边界
- 因实现方便而新增业务语义

## 8. 派生产物标准

派生产物不是规范来源，而是规范消费者。

典型派生产物包括：

- code
- tests
- schema
- migrations
- api contracts
- agent tasks

派生产物 `MUST`：

- 只实现已声明承诺
- 保持字段不变量成立
- 保持功能读写边界成立
- 接受验证层定义的证明要求

派生产物 `MUST NOT`：

- 创造未声明字段
- 创造未声明状态
- 通过实现细节引入新的业务语义
- 把 Promise 缺失当成自由发挥空间

如果实现过程中发现 Promise 不足，正确动作 `MUST` 是修订 Promise，而不是补隐藏逻辑。

## 9. 验证层标准

验证层用来定义如何证明承诺已兑现。

验证层 `MUST` 指向字段层和功能层中的明确承诺。

验证层 `MUST` 定义：

- 要验证的承诺
- 验证方法
- 输入条件
- 预期结果
- 边界场景
- 失败判定

### 9.1 覆盖要求

验证层 `MUST` 覆盖：

- 关键字段不变量
- 关键状态迁移
- 关键读写边界
- 关键拒绝路径

### 9.2 证据要求

验证结果 `MUST` 能映射回具体承诺。

以下内容 `MUST NOT` 作为充分证据：

- “代码看起来没问题”
- “逻辑应该是对的”
- “Agent 说已经实现”

## 10. 定义完成

一个任务只有同时满足以下条件，才算完成：

1. 唯一的 `System Promise` 已定义
2. 字段层、功能层、验证层都已显式存在
3. 派生产物未突破前两层边界
4. 验证已执行
5. 验证结果证明关键承诺成立

因此 Promise 范式中的完成定义是：

`Done = Promise Kept + Evidence Produced`

而不是：

`Done = Code Written`

## 11. 变更规则

以下变更 `MUST` 先修改字段层：

- 新增字段
- 新增状态
- 修改字段语义
- 修改状态迁移规则
- 修改对象不变量

以下变更 `MUST` 先修改功能层：

- 新增行为
- 修改行为前置条件
- 修改读写边界
- 修改拒绝条件
- 修改副作用边界

以下变更 `SHOULD` 同步修改验证层：

- 新增关键不变量
- 新增关键行为
- 修改关键状态迁移
- 修改关键交付标准

## 12. 反模式

以下做法属于 Promise 反模式：

### 12.1 先写实现，再补 Promise

如果 Promise 只是事后整理，它就无法成为 AI Coding 的治理边界。

### 12.2 多份并列真相源

如果字段、功能、验证分别被当成三份并列主文档，系统会失去唯一语义源。

拆分视图 `MAY` 存在，但它们只能是同一 Promise graph 的投影，而不是新的治理源。

### 12.3 用实现细节承载业务语义

如果业务状态只存在于缓存位、布尔旗标、临时分支、注释约定中，那么实现已经越过 Promise。

### 12.4 把验证当成收尾工作

验证层不是实现完成后的装饰，而是 Promise 本身的组成部分。

## 13. 合规检查

一个 `System Promise` 至少应回答以下问题：

1. 这个系统有哪些对象、字段和状态？
2. 这些字段和状态分别代表什么语义？
3. 哪些真相在任何时刻都必须成立？
4. 系统允许哪些行为？
5. 每个行为可以读写什么？
6. 哪些行为和状态是明确禁止的？
7. 要用什么证据证明这些承诺已经兑现？

如果其中任一问题无法回答，该 Promise `SHOULD` 被视为未完成。
